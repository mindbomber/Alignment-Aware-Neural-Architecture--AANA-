# AANA Benchmark Maintainer Review Protocol

This protocol is a maintainer-facing review request for evaluating AANA as an agent-action reliability layer, not as a base LLM benchmark entry.

## Review Goal

Assess whether the AANA pre-tool-call gate should be evaluated as an additional agent safety and reliability track for tool-use benchmarks.

AANA consumes a structured tool-call proposal before execution and returns one of:

- `accept`
- `ask`
- `defer`
- `refuse`

The core claim under review is that structured contracts plus evidence recovery can improve unsafe-action recall while preserving safe-action allow rate, compared with simpler baselines.

## Proposed Input Contract

Each candidate tool call is normalized into this schema before execution:

- `tool_name`
- `tool_category`: `public_read`, `private_read`, `write`, or `unknown`
- `authorization_state`: `none`, `user_claimed`, `authenticated`, `validated`, or `confirmed`
- `evidence_refs`
- `risk_domain`
- `proposed_arguments`
- `recommended_route`

The current JSON schema is in `schemas/agent_tool_precheck.schema.json`.

## Proposed Evaluation Task

For each benchmark trace or benchmark-generated tool call:

1. Extract or synthesize the pre-tool-call contract from the agent trace.
2. Hide the gold label from AANA and all baselines.
3. Run each candidate gate before tool execution.
4. Score the route decision against the benchmark-maintainer label or benchmark rubric.

## Compared Systems

The proposed head-to-head comparison uses:

- Plain permissive agent: always executes.
- Single classifier: unsafe/safe detector without route recovery.
- Prompt-only policy guardrail: text policy prompt without structured contract.
- LLM-as-judge checker: safety checker over the candidate action.
- Structured contract gate without correction or evidence recovery.
- AANA contract gate with verifier/correction/evidence recovery.

## Primary Metrics

- Unsafe-action recall.
- False positives.
- Safe allow rate.
- Route accuracy across `accept`, `ask`, `defer`, and `refuse`.
- Evidence recovery quality where noisy or partial evidence is available.

## Current Evidence

AANA has been tested on two independently sourced public trace datasets with script-generated labels/noise:

- Qwen traces: `zake7749/Qwen-3.6-plus-agent-tool-calling-trajectory`.
- Hermes traces: `NousResearch/hermes-function-calling-v1`.

These runs are useful for engineering validation, but they are not enough for a leaderboard-grade claim because labels and noisy evidence were generated by our evaluation scripts rather than benchmark maintainers or human annotators.

## Maintainer Review Questions

1. Would this be acceptable as an auxiliary safety/reliability track for your benchmark?
2. Which benchmark tasks or trace fields would support the pre-tool-call contract most cleanly?
3. Should labels be human-reviewed, benchmark-rubric-derived, or maintainer-audited from existing traces?
4. Which metric should be primary: unsafe-action recall, safe allow rate, route accuracy, or another benchmark-native score?
5. Would you prefer a pull request adding an adapter/evaluator, a standalone submission artifact, or an issue-discussion first?

## Requested Outcome

We are asking benchmark maintainers to review the protocol before treating AANA results as official evidence. The next step should be a maintainer-approved evaluation path using benchmark-native data and labels.
